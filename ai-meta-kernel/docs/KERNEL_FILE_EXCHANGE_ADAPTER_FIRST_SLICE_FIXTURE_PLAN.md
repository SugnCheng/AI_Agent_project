# Kernel File Exchange Adapter First-Slice Fixture Plan

## Purpose

This document defines the smallest governed fixture strategy for the first acceptable kernel-side file exchange adapter implementation slice.

It is a planning document only. It does not add runtime code, modify kernel contracts, invoke kernel runtime, write response artifacts, write failure artifacts, add live fetching, add scheduler runtime, add report composition, add CI, add package migration, or call external services.

## Planning Decision

Current fixture planning decision:

```text
use_existing_daily_us_core_static_fixture_set_for_adapter_fixture_validation
```

The first acceptable slice should not create a broader fixture universe. It should use the existing `daily_us_core` static envelope, response, and blocking failure examples under:

```text
ai-meta-kernel/examples/file-exchange/
```

The slice should validate fixture shape and adapter scaffold guardrails only. It must stop before kernel intake preparation, P0-P10 runtime invocation, response writing, or failure writing.

## Smallest Static Kernel Input Envelope Fixture Strategy

Use one valid static kernel input envelope fixture:

```text
examples/file-exchange/envelopes/daily_us_core__daily_brief_run__daily_brief__20260423T000000Z__kernel_input_envelope.example.json
```

The envelope fixture should remain:

- `daily_us_core` scoped;
- deterministic;
- macro-produced evidence/context only;
- readable by `read_envelope_artifact`;
- acceptable to `validate_envelope_intake`;
- free of completed canonical task object top-level fields;
- explicit that kernel runtime execution remains deferred.

The envelope fixture should not become a canonical task object, response artifact, or runtime-generated artifact.

## Expected Response Fixture Strategy

Use one static expected kernel response fixture:

```text
examples/file-exchange/responses/daily_us_core__daily_brief_run__daily_brief__20260423T000000Z__kernel_response.example.json
```

The response fixture should remain:

- schema-valid against `meta-layer/TASK_OBJECT_SCHEMA.json`;
- `daily_us_core` scoped;
- an example of a kernel-owned canonical task object artifact;
- static and reviewable;
- not generated by runtime code in the first slice.

The first slice may validate response fixture shape as an expected artifact contract. It must not implement response artifact writing.

## Expected Blocking Failure Fixture Strategy

Use one static blocking kernel failure fixture:

```text
examples/file-exchange/failures/daily_us_core__daily_brief_run__daily_brief__20260423T000000Z__kernel_failure.example.json
```

The failure fixture should remain:

- `artifact_type == "kernel_exchange_failure"`;
- `source_project == "ai-meta-kernel"`;
- `profile_id == "daily_us_core"`;
- `run_mode == "daily_brief_run"`;
- `report_target == "daily_brief"`;
- `blocking == true`;
- free of partial canonical task object content;
- static and reviewable.

The first slice may validate failure fixture shape as an expected blocking contract. It must not implement failure artifact writing.

## Fixtures Needed For The First Slice

The first slice needs exactly these fixture classes:

| Fixture class | Required fixture count | Current path |
| --- | ---: | --- |
| Kernel input envelope | 1 | `examples/file-exchange/envelopes/` |
| Expected kernel response | 1 | `examples/file-exchange/responses/` |
| Blocking kernel failure | 1 | `examples/file-exchange/failures/` |

No additional profiles, regions, report targets, runtime artifacts, generated artifacts, or committed failing fixtures are needed for the first slice.

## First Slice Local Validation Requirements

The first slice must validate locally that:

1. the envelope fixture exists and parses as a JSON object;
2. the envelope fixture satisfies the current envelope intake contract;
3. the envelope fixture does not include completed canonical task object top-level fields;
4. the response fixture exists and parses as a JSON object;
5. the response fixture validates against `meta-layer/TASK_OBJECT_SCHEMA.json`;
6. the failure fixture exists and parses as a JSON object;
7. the failure fixture is blocking;
8. the failure fixture does not include a partial canonical task object;
9. `prepare_kernel_intake` remains fail-closed;
10. `invoke_kernel_runtime` remains fail-closed;
11. `write_response_artifact` remains fail-closed;
12. `write_failure_artifact` remains fail-closed.

The existing local helpers already cover most of this surface:

```text
validation/kernel_file_exchange_fixture_checks.py
validation/kernel_file_exchange_adapter_scaffold_checks.py
```

A future slice may refresh or narrow those checks, but it must not broaden into runtime behavior.

## Blocked In The First Slice

The first slice must not add:

- runtime adapter invocation;
- P0-P10 runtime execution;
- envelope-to-P0/P1 intake implementation;
- canonical task object generation from envelope evidence;
- response artifact writing;
- failure artifact writing;
- runtime artifact directory scanning;
- artifact polling or watcher behavior;
- retry/backoff behavior;
- cleanup automation;
- CLI command design;
- multi-profile runtime expansion;
- live fetching;
- scheduler runtime;
- report composition;
- archive/export automation;
- CI behavior;
- package migration;
- external service calls;
- kernel contract modifications;
- macro-side production of kernel response artifacts.

## Drift Requiring A Governed Pass

The following changes require a governed pass before implementation:

- adding additional fixture classes;
- adding additional profiles beyond `daily_us_core`;
- changing fixture filenames or naming semantics;
- changing envelope required fields;
- allowing canonical task object fields in envelope fixtures;
- changing response fixture schema expectations;
- changing failure fixture shape;
- allowing failure fixtures with `blocking == false`;
- using generated runtime artifacts as committed fixtures without review;
- implementing any reader/writer/runtime behavior beyond current scaffold validation.

## Recommended Next Phase

Implement a `Kernel-Side Adapter Fixture Validation Output Contract Pass`.

That pass should snapshot the output contract for the first-slice fixture validation surface, using the existing static fixture set and current scaffold guardrails, while still avoiding runtime code, response/failure writers, CLI, CI, scheduler behavior, live fetching, report composition, package migration, and actual handoff execution.
